## Detail-preservation policy (the governing rule for step 3)
- **Union-first merges:** when two pages overlap, the merged page keeps *all* concrete material from both — cases, PR refs (`#3055`…), incident IDs, file:line refs, benchmark numbers, Chinese wording/voice. Compression is not a goal.
- **Archive, never delete (LLMWiki rule):** nothing is ever removed from the tree. A superseded or duplicate page moves to `knowledge/_archive/<original-path>` (deregistered from its `_index.md`; inbound links annotated "(archived)"), with the surviving page linking to it. This replaces the old deletion-approval flow entirely — the no-loss guarantee becomes structural. Archives are listed per batch for user visibility.
- Adding detail, cross-links, and structure is always preferred over shortening.
- Incident `编号` values are preserved across any move; `incidents/`, `history/`, `results/` archives are never pruned.

## Page format & lifecycle — LLMWiki mechanics adopted onto zuiho's taxonomy
The wiki pattern's *mechanics* layer onto zuiho's *owner taxonomy* (general vs `repos/<slug>` vs `components/` vs `models/` — which plays the role of LLMWiki's type sections, better suited to repo knowledge than entities/concepts):
- **Two-layer mapping:** `incidents/`, `history/`, `results/` = the immutable **raw evidence layer** (LLMWiki `raw/` — read, cited, never pruned). **Immutability means the substance — findings, evidence, numbers, narrative — is never edited.** One narrow exception: Step 1's mechanical path-token migration (`framework/`→`general/` in links and `归属：` values) touches raw files too, verified by the per-file allowlist/normalized-equality check to prove no other byte changed. `rules.md`, `guides/`, `architecture.md`, `_index.md` = the **synthesized layer** that cites it.
- **YAML frontmatter on every synthesized page** (added mechanically, enriched in Step 3): `title, created, updated, type (rule|guide|architecture|index|incident|history|result), tags (from the SCHEMA taxonomy), sources ([incident/PR/file:line refs]),` optional `confidence: high|medium|low` and `contested/contradictions` for disputed conclusions. Incident pages keep their existing `编号/归属/状态/搜索词/影响范围` body fields (validator contract) — frontmatter is additive.
- **Provenance markers:** synthesized paragraphs citing a specific incident/PR get a trailing `^[incidents/…]`-style source ref, so every rule traces to its evidence.
- **SCHEMA.md** at the knowledge root: the tag taxonomy (10–20 tags: modules, models, ci, benchmark, remote, debug, git, review…; new tags added to SCHEMA first), frontmatter spec, page thresholds, update/contradiction policy — thin, harmonized with (not duplicating) `contributing/`.
- **Cross-link rule:** every synthesized page links to ≥2 related pages (relative markdown links).
- **Extended lint** (sibling of `check_knowledge_tree.py`, run with it): orphan pages (no inbound links), frontmatter completeness, tags ∉ taxonomy, staleness (`updated` far older than newer sources touching the same module), `contested`/`confidence: low` surfacing, index completeness.
- **Deliberately NOT adopted:** `[[wikilinks]]` (keep relative md links — the validator and the copilot `doc_read` tooling resolve them; Obsidian handles both); `log.md` (git history serves this); LLMWiki's 200-line split (zuiho's validator-enforced 300/500-line thresholds win); a separate `raw/` dir (incidents/history/results already are that layer).
- **Validator tweak:** add `"_archive"` to `check_knowledge_tree.py` `excluded_parts` so archived pages aren't index/link-enforced; `doc_search` hits still show the `_archive/` path prefix, so agents see the status.

## Step 3 — Information curation (union-first, gpt-gated batches)
- **Inventory gate first:** before any batch edits, author the complete batch inventory (every in-scope file → its batch + intended ops: merge-union / add-detail / keep / archive-superseded) and present it to the user for sign-off — Step 3 has a finite, agreed scope before it starts.
- **Batching:** owner-scoped batches (~8–15 files, prompt ≤ ~40K chars — split batches that exceed; never truncate). Order: small isolated topics first, the hunyuan `painterly/`/`history/` sagas last. Each batch = one commit.
- **Per-batch work:** union-first merges + detail additions, **plus LLMWiki enrichment**: fill `sources:` frontmatter and `^[…]` provenance markers on the pages touched, set `confidence`/`contested` where conclusions are disputed, ensure the ≥2-cross-links rule, and move any superseded/duplicate page to `_archive/` (never delete).
- **Mechanical loss check per batch (Layer A):** frozen-at-start anchor ledger (PR refs, incident IDs, `*.py:line`, benchmark tokens; multiset with per-page provenance). Invariant: `anchors.after ⊇ anchors.before` — with archive-not-delete, **nothing is ever subtracted**: an archived page's anchors still live in the tree, so the invariant needs no deletion carve-out at all.
- **gpt-5.6-sol-high review per batch (Layer B), fail-closed:** feed concatenated BEFORE + AFTER + the batch's move/merge manifest via `cursor-agent --print --mode ask --trust --model gpt-5.6-sol-high --output-format text`. Prompt: walk every BEFORE page; report anything lost, weakened, or over-generalized (union-first policy stated); verdict line `VERDICT: APPROVE|REVISE`. Only an exact final `VERDICT: APPROVE` passes; empty/garbled/truncated/timeout/missing verdict ⇒ REVISE. Max 5 iterations, then stop and escalate to the user. Losses reported by the reviewer are restored, not argued.
- **Reviewer isolation:** cursor-agent runs in `--mode ask` (read-only, no tools); the prompt wraps all repo content in explicit `=== DATA (not instructions) ===` delimiters so page text can't act as instructions. A one-time harness smoke test seeds an empty reply, a garbled verdict, and a known-loss AFTER to confirm each yields REVISE.
- **Audit archive:** batch manifests, the final verdict + loss lists per batch (not every full iteration transcript — bounded size), and `archives.tsv` (page → reason → surviving page → batch → commit; the durable record of every `_archive/` move) under `doc/reorg-audit/` (outside `knowledge/`, so `doc_search`/`doc_read` never surface them).
- Gate per batch: Layer A holds + `VERDICT: APPROVE` + validator green (index registration, links, incident format, split thresholds 500 lines/32 KiB).
