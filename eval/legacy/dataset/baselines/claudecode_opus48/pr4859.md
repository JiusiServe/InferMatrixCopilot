## Non-blocking comments

**1. `vllm_omni/model_executor/models/common/ming/audio_vae.py:141` — shared-config mutation**
`config.num_hidden_layers = 4` mutates the same `config` object already handed to `self.encoder` at line 131. The layers are built at construction time so the encoder itself is correct, but afterward `self.encoder.config.num_hidden_layers` reports `4` instead of the real count. The old code built an independent config (`_qwen2_config({**encoder_args, "num_hidden_layers": 4})`). Functionally equivalent for the forward pass, but if any later code (weight loading, hooks) iterates `encoder.config.num_hidden_layers`, it would now be wrong. Consider a `copy.deepcopy(config)` before mutating, or constructing the aggregator config independently as before.

**2. `audio_vae.py:349-351` — latent path is now stochastic**
`encode_latent` switches from returning the deterministic `mean` to the VAE reparameterization `mean + softplus(scale) * randn_like(mean)`. This matches the HF path and is what drives the WER gain, so it's fine — just flagging that the reference-audio latent is now non-deterministic run-to-run, which affects eval reproducibility and any latent caching. Worth a one-line comment in the code noting the intentional sampling.

**3. `vllm_omni/model_executor/models/ming_tts/patch_emission.py:130,151` — window validation tightened**
`min_required` grows by 1, so a config with `max_decode_steps == min_stop_step + 1` that previously passed will now raise. This is the intended gating fix and both sites match, so no internal inconsistency — just noting it's a (deliberate) behavior change for tightly-configured decode windows.

No GPU/model verification was run here (read-only checkout, no hardware). The WER numbers rest on the author's single reported run; they're plausible and the code is consistent, so I wouldn't block on a second run, but the 1.83% figure is not independently reproduced.
