## Specific comments

**1. (Medium — correctness) Integer `version=1/2` never pins anything; it's silently swallowed.**
`vllm_omni/diffusion/attention/backends/flash_attn_hub.py:31-39` (merged main; in the PR diff this is the inline `get_kernel(..., version=1)` → `version=2` → default cascade in each `__init__`).
```python
for version in (1, 2, None):
    try:
        if version is not None:
            return get_kernel(repo_id, version=version)   # version expects a PEP 440
        return get_kernel(repo_id)                         # specifier string, not an int
    except Exception as exc:
        logger.info(...)                                   # swallowed at INFO
```
`kernels.get_kernel`'s `version` parameter takes a PEP 440 specifier *string* (e.g. `">=2.0.0,<3.0.0"`), not an int. As written, both the `version=1` and `version=2` attempts raise and are caught, so every load falls through to the unpinned default kernel — and prints two spurious `Failed to load ... version 1/2` INFO lines. Since this PR's entire premise is *deterministic* kernels matching the training side, silently resolving "whatever is latest" undermines the goal. Please either pass a real specifier string / pinned revision, or drop the loop and document that the default (latest) is used. The broad `except Exception` also masks genuine failures (network/auth) — worth narrowing.

**2. (Low — maintainability) `FlashAttention3HubImpl`/`FlashAttention3HubBackend` are ~95% copy-paste of the FA2 classes.**
`flash_attn_hub.py:264-406` duplicate `:90-241` almost verbatim — the only differences are the repo-id string (`flash-attn3` vs `flash-attn2`) and one comment. The loader was already factored into `_get_hub_module`; the two Impl classes should share a base class parameterized by `repo_id` (and the class-level `get_name`/head sizes). Two copies will drift.

**3. (Low — reliability) Masked/varlen paths call `self.flash_attn_varlen_func` unconditionally, but `__init__` only guarantees *one* of the two funcs is non-None.**
`flash_attn_hub.py:122-123` accepts a build that exposes only `flash_attn_func` (varlen `None`). But `_forward_varlen_masked` (`:161`, `:326`), `_forward_varlen_dense` (`:183`, `:348`), and `_flash_wrapper`'s else branch (`:133`, `:298`) all call `self.flash_attn_varlen_func(...)` — a masked or `full_attn_spans` request against such a build raises `TypeError: 'NoneType' object is not callable`. Either require `flash_attn_varlen_func` to be present, or fall back to `flash_attn_func` for the masked path.

**4. (Nit) Redundant `if/elif` in the CUDA fallback.**
`vllm_omni/platforms/cuda/platform.py:129-138` — the `FLASH_ATTN_HUB` and `FLASH_ATTN_3_HUB` branches do exactly the same thing (identical warning, both set `backend_upper = "FLASH_ATTN"`). Collapse to a single branch. (The musa/npu/rocm/xpu platforms already handle both in one branch.)

**5. (Evidence) The core numerical-consistency claim is not tested, and FA3 execution is untested in CI.**
`tests/diffusion/attention/test_kernels_hub.py:88-91` only asserts Hub-FA output ≈ local SDPA within `1e-2`. That verifies "the kernel runs and is roughly attention," not the PR's actual claim of parity with the kernels diffusers loads during training. Consider an equivalence assertion against the same `kernels` module loaded the diffusers way. Also, `test_kernels_hub_execution` is marked `hardware_test(res={"cuda":"L4"})` — L4 is sm_89, so the FA3 block is `pytest.skip`-ed; the FA3 execution path only has the manual E2E screenshot in the PR body as evidence. Worth noting explicitly.
