## Review: PR #4977 — `[Diffusion] Improve HuggingFace hub kernel loading in flash_attn_hub`

**Recommended verdict: APPROVE** (non-blocking notes below)

Clean, well-scoped change. It hoists the per-`__init__` Hub kernel load into a process-level, lock-guarded cache (`_get_hub_module`), so each of `flash-attn2` / `flash-attn3` is fetched once per process instead of on every `FlashAttention*HubImpl` construction. Since diffusion builds one attention backend per layer, this is a real startup win. The version-fallback semantics (`v1 → v2 → default`) are faithfully preserved by the `for version in (1, 2, None)` loop, and load failures are correctly *not* cached (the exception propagates before the dict assignment on `flash_attn_hub.py:45`), so a transient failure won't poison the cache for the rest of the process.

2. **`flash_attn_hub.py:43-46` — lock is held across the network/disk load.** `_get_hub_module` holds `_hub_lock` for the full duration of `_load_hub_module` (which does Hub I/O). That correctly prevents a duplicate concurrent load of the *same* repo, but it also serializes an in-flight `flash-attn2` load against a concurrent `flash-attn3` load. This is a one-time startup cost and harmless in practice, so I'd only bother if worker init concurrency matters — a per-repo lock (or double-checked locking) would let the two repos load in parallel.
