# Review: PR #4762 — [Core / Bugfix] Add Mechanism for Endpoint Rejection

**Repo:** `vllm-project/vllm-omni` · **Author:** alex-jw-brooks · **Status:** MERGED (`bug`, `high priority`) · reviewed post-merge at `/rebase/vllm-omni`

## Summary

Solid, well-scoped bugfix. It stops a class of server crashes where sending a request to an endpoint a model can't support (e.g. `/v1/completions` for the Qwen3-Omni thinker→talker path, which needs ChatML markers) takes down the whole instance. The mechanism — a per-`PipelineConfig` tuple of `EndpointRestriction`s that swap the route for a 400-returning handler at server-init — is clean and extensible, and the always-on `UNSUPPORTED_ROUTES` (batched chat completions) is a reasonable guardrail. The bulk of the diff is a genuine improvement: `config_factory.py` is refactored so model-type inference, HF-config loading, and pipeline resolution are separate cached methods, and endpoint-restriction resolution reuses the exact same `get_pipeline_config` path as stage resolution — so restrictions can't diverge from the pipeline that actually gets built.

## Non-blocking comments

**1. `vllm_omni/config/config_factory.py` — `trust_remote_code` default silently flipped `True` → `False`** (`create_from_model` signature `trust_remote_code: bool = False`, and `entrypoints/utils.py` `load_stage_configs_from_model` now passes `cli_overrides.get("trust_remote_code", False)`).

The old `create_from_model` used `cli_overrides.get("trust_remote_code", True)` — i.e. when the key was absent, auto-detection loaded the HF config *with remote code trusted*. The refactor defaults it to `False`. For a custom-code model whose `get_config` only succeeds with `trust_remote_code=True`, `get_hf_config` now returns `None`, so `hf_config`-dependent resolution (callable resolvers, the `hf_architectures` intersection fallback) degrades to the config.json/`model_index.json`/name-matching fallbacks. This is arguably a *correctness/security improvement* (matches vLLM's own default), but it's an unmentioned behavior change bundled into an endpoint-rejection PR. Worth confirming it's intentional and that no registered model relied on the old `True` default for detection.

**2. `vllm_omni/config/endpoint_policy.py:52-62` — `build_rejection_handler` reads `status_code=error.error.code` directly.** Elsewhere in `api_server.py` (`_error_response_to_json_response`, line ~362) the code defensively falls back to `HTTPStatus.BAD_REQUEST` when `err.error.code` is `None`. Here it's used raw. Tests confirm `create_error_response(message=...)` yields `400` today, so this is fine in practice — just slightly more fragile than the sibling pattern if that default ever changes. Minor.

**3. `vllm_omni/entrypoints/openai/api_server.py:~513` — `else` branch skips *all* restrictions, including `UNSUPPORTED_ROUTES`.** When `engine_client` lacks `endpoint_restrictions`, the code only logs a warning and calls nothing — so the always-on batched-chat-completions guard (`UNSUPPORTED_ROUTES`) is *also* skipped, leaving `/v1/chat/completions/batch` able to 500. `AsyncOmni` always sets the attribute now, so this is latent, but since `UNSUPPORTED_ROUTES` is meant to be unconditional, calling `shutdown_unsupported_routes(app, ())` in the else branch (rather than skipping) would be more faithful to intent.

**4. `config_factory.py` — `@functools.cache` on `get_hf_config` / `try_infer_model_type` / `get_pipeline_config` never invalidates and memoizes `None`.** Benign for a server (configs load once), and the new autouse `clear_config_factory_caches` fixture handles it — but that fixture lives only in `tests/test_config_factory.py`. Other modules that touch the factory (e.g. entrypoint tests) don't clear it, so a transient `get_config` failure or overlapping model string could leak state across those tests. Low risk given they mostly monkeypatch, but worth keeping in mind if flakes appear.
