## Non-blocking comments

**1. `tests/helpers/assertions.py:658` — containment fallback now applies to *all* speech tests, not just opt-in ones.**
Previously the leniency (ASR escalation) was opt-in via `transcript_escalation_model`; strict behavior was preserved otherwise. This new containment fallback runs unconditionally before escalation, so it globally loosens every caller of `_assert_transcript_matches`. For a short expected phrase this can accept a wrong-but-longer transcript, e.g. expected `"hello"` vs transcript `"hello world goodbye"` passes (`small_word_delta`: 3 ≤ 1+2, and `"hello"` is contained). Consider gating it behind a per-request opt-in like escalation, or tightening `small_word_delta` for very short expected phrases. Low severity — it's test-gate tuning, and the unrelated-transcript case is still rejected (covered by `test_short_transcript_unrelated_text_still_fails`).

**2. `tests/helpers/assertions.py:60` — `short_text` uses `min(...)`, making the "short clip" gate effectively always true.**
`short_text = min(len(transcript_clean), len(expected_clean)) <= 15` is satisfied whenever *either* string is ≤15 chars — since `expected` is the (usually short) test input, this is nearly always true regardless of the actual clip/transcript length. The real guards are `small_word_delta` + containment. If the intent was "the clip is genuinely short," `len(transcript_clean) <= 15` (or a word-count cap on the transcript) would express that; as written the `short_text` term does little. Cosmetic, not a correctness issue.
