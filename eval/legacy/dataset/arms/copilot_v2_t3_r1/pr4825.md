`vllm_omni/diffusion/lora/manager.py:375` [minor] ŌĆö This PR adds `'unet'` to `default_components` to enable LoRA discovery on SDXL pipelines. However, SDXL's pipeline class already declares the denoiser via `_dit_modules: list[str] = ['unet']` (pipeline_sdxl.py:46). The hardcoded tuple duplicates this declarative metadata; consider deriving `default_components` from `self.pipeline._dit_modules` (using the first segment of each dotted path) to avoid maintaining two synchronized lists. If that's not in scope, add a TODO comment referencing `_dit_modules` so future maintainers are aware of the relationship. (evidence: grep _dit_modules in vllm_omni/diffusion/models shows every pipeline declares it; StableDiffusionXLPipeline at pipeline_sdxl.py:46 sets it to `["unet"]` ŌĆö the same value this PR hardcodes.)

`vllm_omni/diffusion/lora/manager.py:375` [minor] ŌĆö This PR adds `'unet'` to `default_components`, causing the LoRA manager to scan `self.unet` on every pipeline that has that attribute, not just SDXL. For example, `flux2_klein` and `ovis_image` pipelines reference `self.unet` in docstrings but declare `_dit_modules = ['transformer']` (denoiser is a transformer, not unet). The extra scan is likely harmless but unintended scope expansion. Consider restricting `unet` scanning to pipelines that declare `'unet'` in `_dit_modules`, or have SDXL opt in via `_lora_components`. (evidence: grep 'self\.unet' vllm_omni/diffusion/models/flux2_klein/ shows docstring references to `self.unet.config.sample_size`; grep 'self\.unet' vllm_omni/diffusion/models/ovis_image/ shows same pattern; both pipelines' `_dit_modules` (grep output) = `['transformer']`.)
