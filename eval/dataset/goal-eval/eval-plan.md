# Evaluation plan v3 — quality/cost/latency vs baseline (goal task #23)

## Context
W1-W7 landed (uncommitted) in /rebase/vllm-omni-copilot. This plan measures the quality gates and reports quality+cost+latency together. Metric definitions REUSED unchanged (eval/dataset/judge_val.py: 7 dims, blind pairwise vs the recorded CC+Opus arm, 3 judge replicates, Sonnet-5 judge). The recorded baseline is never rerun. The evaluation is **retrospective synthesis under thread visibility** (issue arms read the resolution thread, as every arm always has), NOT blind pre-resolution maintenance — stated verbatim in the report.

## Arms — all on the val split (5 PR + 5 issue); every generation replicate gets a unique ARM dir and a unique JUDGE_OUT
- **A0**: existing `arms/copilot_v2_t4_r{1,2,3}` val outputs, **re-judged now** under the identical current protocol (never substituted by old/test numbers; missing-dim substitution is forbidden).
- **A1**: current tree, `MOA_WHEN=off`, `review_depth=auto`, `PR_CONTEXT_MODE=no_discussion` → `arms/copilot_v3_r{1,2,3}` (3 generation replicates, precommitted).
- **A2**: A1 + `MOA_WHEN=always` (live LLM_MIXTURE members, `moa_max_usd=1.50`) → `arms/copilot_v3_moa_r{1,2,3}` (**all 3 replicates precommitted — no optional stopping**).

## Judging & validation (machine-enforced)
- Every (arm, generation replicate) judged with `SPLIT=val ARM_A_DIR=<arm_r_k> JUDGE_OUT=judgments/val_<arm>_r<k>` — unique dirs so judge_val's skip-existing behavior can never mix arms.
- A new `eval/dataset/validate_replicate.py` gates every replicate BEFORE aggregation: the exact 30 expected verdict FILENAMES (10 stems × r1-r3), each verdict schema-validated per its dimension set (PR: recall/precision/actionability/gap_hit; issue: correctness/grounding/completeness), values in [0,1] or the winner/margin enum, `_blinding` map present and consistent, judge-replicate index matching the filename; plus 10/10 arm outputs with rc==0 (successful, not merely present), and for every PR item the run's checkout note must assert `PR-TIME TREE` with the expected head SHA (the fetch step's live-checkout fallback returns success — rc alone is insufficient). Incomplete/invalid ⇒ judge retried; still bad ⇒ replicate invalid and re-run.
- Pairing: for each dim, score(arm) = mean over its 3 generation replicates × 3 judge replicates of the pairwise rubric vs the SAME fixed CC+Opus outputs; Δ(A1−A0) computed per dim on these unrounded means, with per-item paired deltas + sign counts reported.

## Pre-campaign implementation (blocking; with unit tests, before any paid run)
1. Extend `aggregate_costs.py`: attempt-inclusive TOKENS/USD/by_role (summed over every attempt's run dirs + invocation records, not just the final run) — these span sums are **service time/spend, never called latency**; the ONLY latency metric anywhere (code, fixtures, report) is the outer end-to-end `wall_s`; a `--baseline --split val` mode reading `baselines/*.cost.json` (`cost_usd`, tokens, wall_s) that REQUIRES a split argument and validates the EXACT 10 expected val item stems (the baseline dir mixes val/train/test artifacts; wrong or missing items ⇒ hard error). Unit-test on synthetic fixtures.
2. `validate_replicate.py` (above), unit-tested on synthetic judgment dirs.
3. A0 limitation acknowledged: its artifacts predate invocation records — A0 cost basis = its recorded final-run metrics, labeled "final-attempt-only (retries not reconstructable)"; A1/A2 report BOTH bases so the comparison to A0 uses the matching basis.

## Procedure
0. **Provenance snapshot BEFORE any val generation** (`eval/dataset/provenance_val.json`): the cleanroom-script patch and all pre-campaign tooling land FIRST, then the snapshot: {HEAD sha, sha256(`git diff --binary HEAD`), NUL-safe untracked-SOURCE digest (`git ls-files -zo --exclude-standard -- src playbooks 'eval/dataset/*.py' 'eval/dataset/*.sh' | xargs -0 sha256sum | sort`) — GENERATED artifacts (arms/, judgments/, invocations, ledger, provenance files themselves) are explicitly excluded so the campaign cannot self-invalidate, pip freeze digest, resolved tier models, mixture member labels, MODEL_PRICES, and the full effective arm env (every OMNI/MOA/PR_CONTEXT/REVIEW knob)}. Asserted UNCHANGED before every later stage (A2, judging, final); any drift ⇒ stop. Each arm dir gets a `provenance.json` binding it to this digest + a sha256 manifest of its own outputs; `judge_val.py` gets a metadata-only patch recording ARM_A_DIR and the arm content hash into each verdict file (its hardcoded `copilot_v2` label cannot detect a wrong arm; scoring logic untouched, change documented).
1. Preflight: doctor green; canary item (pr4816) per replicate config before the full batch; verify from the canary's run evidence that pr_context is present WITHOUT a discussion section.
2. Generate A1 r1-r3, then A2 r1-r3 (sequential; item-private RUN_ROOTs; per-item bound = runner's 4 attempts × 3000s). **Fresh directories per replicate attempt**: every generation replicate (and any re-run after invalidation, and the canary) uses a NEW `ARM_OUT` dir (`_attempt2` suffix on re-runs) — run_copilot_arm.py skips existing non-empty outputs, so a failed canary/partial replicate is never resumed into; invalid dirs are renamed `.invalid` and excluded from aggregation.
3. Judge: A0 r1-r3 (re-judge), A1 r1-r3, A2 r1-r3 → 9 × 30 = 270 verdicts; +90 if the final confirmation runs. Budget: ~$0.20/verdict ⇒ ~$54 judging + ~$72 with final; generation ~$10-25. **Campaign ceiling $150 INCLUDING the conditional final confirmation, enforced preventively**: an append-only spend ledger (`eval/dataset/campaign_ledger.jsonl`) records EVERY paid event as it happens — generation attempts (canaries, invalid dirs, and FAILED invocations that produced no artifact included) and judge calls incl. retries; before each paid batch the runner atomically reserves a CONSERVATIVE HARD MAXIMUM (per-item worst case: 4 attempts × 2× the largest observed per-attempt cost; judging: verdicts × $0.30) and refuses the batch if reserved+settled would exceed the ceiling; actual spend reconciles the reservation after the batch (settled ≤ reserved by construction).
4. Cost/latency: extended aggregate_costs.py on every arm + the baseline dir → mean/median/p95 of end-to-end `wall_s` (THE latency metric — never a sum of nested/parallel spans, which double-counts concurrent MoA/lens time), USD (both bases), tokens, per-phase step durations and per-role span sums both labeled **service time, not wall latency**, TTFR, cost-per-successful-task; MoA/planner overhead from cost.by_role (service-time/token framing).
5. **Promotion gate (vs A0, paired on val, unrounded)**: PR recall AND issue completeness Δ(A1−A0) ≥ +0.03 (the min meaningful effect); no other gated dim (precision, actionability, correctness, gap_hit, grounding) lower than A0 by > 0.02; **cost gate is matched-basis: A1 FINAL-ATTEMPT mean cost ≤ 1.5 × A0 final-attempt mean cost** (final-vs-final; the attempt-inclusive number is reported separately and, being ≥ the final-only number, can only make the reported picture more conservative, never the gate laxer). A2 promotes MoA default-on only by the same rule vs A1. **A2 validity (executable)**: auto-depth LIGHT PRs never reach the ensemble, so MoA cannot engage there BY DESIGN — validate_replicate asserts, for every A2 item that ran the ensemble path (agent_ensemble event present), a `moa_dispatch` with ≥2 expected member labels AND ≥1 successful member-model llm SPAN (attr.model ∈ member set, role=moa_member — dispatch alone proves nothing if every member dropped); light items are recorded `moa: n/a (light)`; the A2-vs-A1 comparison is reported overall AND on the MoA-engaged subset; member drops/fallbacks counted and reported. A replicate whose every eligible item is degenerate is excluded and re-run ONCE; still degenerate ⇒ HARD FAIL of the A2 arm (reported as such, no silent A1 duplicate).
6. **External comparison (vs CC+Opus, report-only)**: per-dim means vs the recorded arm + real recorded cost distribution (per-item cost.json over the same items). "Pareto-consistent" claimed ONLY if every reported dim ≥ the baseline's AND cost ≤ — else stated as an explicit trade-off. Effects < +0.03 are described as consistent with noise, never as wins.
7. **Final confirmation (once, only if the promotion gate passes)**: patch `eval/dataset/run_test_cleanroom.sh` FIRST — pin the A1 env (MOA_WHEN=off, PR_CONTEXT_MODE=no_discussion, REVIEW_DEPTH=auto), remove every `|| true`, fail closed on <30 verdicts or <10 rc==0 items per replicate, write `TEST_CLEANROOM_V3.md`, snapshot into the output {content digest covering TRACKED AND UNTRACKED evaluated source = sha256 over `git diff` + `git ls-files -mo --exclude-standard -- src eval playbooks | sort | xargs sha256sum` (untracked W1-W7 files included), resolved tier models, mixture member labels, MODEL_PRICES, all pinned env knobs, pip freeze digest}, and persist the numeric cost/latency aggregates BEFORE purging artifacts. Labeled "repeated confirmation on the frozen test split" (it was judged once before and informed targets — not an unbiased headline).
8. Report `doc/EVAL-goal-report.md`: evidence matrix, per-dim val tables (A0/A1/A2 ± sd, paired deltas, sign counts) + test confirmation, both cost bases, overhead breakdown, protocol caveats (retrospective-synthesis framing, judge κ variance, baseline merged-PR contamination note), gate outcomes. No combined score.

## Failure honesty
Gates fail ⇒ report honestly, keep UX/instrumentation wins, list next levers (B8 repo-map), do NOT run the test split.

## Review-round-6 refinements (folded; binding on the tooling)
1. Provenance digests use zero-terminated pipelines end-to-end (`... | xargs -0 sha256sum | sort -z | sha256sum`); the final-confirmation snapshot uses the SAME source-only path set (src, playbooks, eval/dataset/*.py|*.sh) and excludes provenance/manifest/ledger outputs themselves.
2. **No raw env persisted**: the provenance records normalized NON-SECRET settings only — member `model@host` labels, knob values from an allowlist; `LLM_MIXTURE` raw value (may contain `api_key`) is never written; anything unrecognized is redacted.
3. `validate_replicate.py` ENFORCES (hard error, not log) the verdict-file ARM path + arm content hash against the arm's provenance.json, and validates PR-time checkout against an independently frozen `expected_pr_heads.json` map (captured at snapshot time via gh, not read back from the runs it validates).
4. A2 validation covers issue items too (issue MoA has no agent_ensemble event): under MOA_WHEN=always every issue item and every non-light PR item must show `moa_dispatch` + ≥1 successful member span; span checks compare `attr.model` to raw member model names (dispatch labels are model@host).
5. Reservations use derivable hard maxima, not estimates: generation per attempt = the runner's max_tokens/iteration ceilings × price (the same arithmetic as the MoA ledger), judging per verdict = judge max_tokens × Sonnet price; the conditional final-confirmation budget is EARMARKED at campaign start before any discretionary spend.
6. `.invalid` dirs are excluded from quality/final-attempt statistics but their spend/tokens ARE included in attempt-inclusive totals (never undercounted).
7. The final confirmation runs THE PROMOTED configuration: A1 if MoA does not promote, the A2 config if it does — the cleanroom env pin is parameterized accordingly.
