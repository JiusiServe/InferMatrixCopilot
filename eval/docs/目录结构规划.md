# vLLM-Omni Copilot Eval 目录与 Benchmark 数据规划

## 1. 文档目的

本文定义 `vllm-omni-copilot` 项目的统一评测目录结构，以及 PR Review Benchmark 的正式数据格式。

规划目标：

1. 所有评测相关代码、数据、配置和运行产物统一放在 `eval/` 下，避免项目根目录出现过多目录；
2. 支持 `pr_review`、`issue_answer`、`bug_fix`、`pr_assignment` 等多种评测任务；
3. 明确区分评测代码、Benchmark 数据、配置和运行产物；
4. 明确区分 Agent 可见信息、GT 和 Judge 私有信息；
5. 支持 Benchmark 版本冻结、完整性校验和结果复现；
6. 保持不同评测任务之间相互独立，同时允许复用公共基础设施。

------

# 2. 总体设计原则

## 2.1 按职责组织顶层目录

`eval/` 下只设置以下几类顶层目录：

```text
common
tasks
configs
data
runtime
```

分别表示：

| 目录       | 用途                                 |
| ---------- | ------------------------------------ |
| `common/`  | 不同评测任务共享的基础代码           |
| `tasks/`   | 各类评测任务的具体实现               |
| `configs/` | Agent、Judge、工具权限等运行配置     |
| `data/`    | Seed、正式 Benchmark 和版本清单      |
| `runtime/` | 缓存、临时 workspace、运行结果和报告 |

这样可以避免随着评测任务增加，在 `eval/` 根目录不断增加互不相关的目录。

------

## 2.2 评测代码和评测数据严格分离

以下内容不能混在同一个目录中：

```text
评测程序代码
Benchmark GT 数据
GitHub 历史证据
Agent 运行结果
报告
仓库缓存
```

特别是：

- `eval/tasks/pr_review/benchmark/` 是 Benchmark 构建和加载代码；
- `eval/data/pr_review/benchmarks/` 才是正式 Benchmark 数据；
- 不能因为二者都叫 Benchmark，就将 YAML 数据放进 Python 源码目录。

------

## 2.3 每种任务使用独立命名空间

不同任务分别放在：

```text
eval/tasks/pr_review/
eval/tasks/issue_answer/
eval/tasks/bug_fix/
eval/tasks/pr_assignment/
```

不同任务可以拥有不同的：

- Benchmark Item Schema；
- Agent 输入协议；
- Agent 输出协议；
- Judge Rubric；
- 指标；
- Runner；
- 报告。

公共能力统一放在 `eval/common/`，不能通过让一个任务直接依赖另一个任务的内部模块来实现复用。

------

## 2.4 运行产物不提交到 Git

以下目录属于运行时数据，应加入 `.gitignore`：

```text
eval/runtime/cache/
eval/runtime/workspaces/
eval/runtime/runs/
eval/runtime/reports/
```

正式 Benchmark 可以进入版本管理，但正式 Test Set GT 应存储在私有环境中，不能提交到公开仓库。

------

# 4. 各类目录职责

## 4.1 `eval/common/`

存放跨任务共享、与具体业务语义无关的代码，例如：

- YAML 和 JSON 文件读写；
- SHA-256 计算；
- 通用版本号类型；
- LLM Backend 接口；
- token usage 结构；
- 重试策略；
- 通用统计函数；
- Run Metadata 基础结构。

不应放入：

- PR Finding；
- PR Verdict；
- Issue Answer Category；
- Bug Fix Patch；
- 某一种任务特有的 Rubric。

判断原则：

> 只有至少两个评测任务能够在不理解彼此业务语义的情况下直接复用的能力，才能进入 `common/`。

------

## 4.2 `eval/tasks/`

存放每种评测任务的代码实现。

例如：

```text
eval/tasks/pr_review/
```

负责：

- PR Review Benchmark Schema；
- PR Review Agent 输入和输出；
- PR Review Judge；
- PR Review 指标；
- PR Review 报告；
- PR Review CLI。

未来其他任务应使用同样的任务边界：

```text
eval/tasks/issue_answer/
eval/tasks/bug_fix/
eval/tasks/pr_assignment/
```

不同任务之间不能直接导入对方的内部实现。

------

## 4.3 `eval/configs/`

存放可版本化的运行配置，包括：

- Agent 配置；
- Judge 模型配置；
- temperature；
- prompt version；
- Tool Policy；
- Jury 组成；
- 重试次数；
- 第二轮裁决配置。

配置按照任务划分：

```text
eval/configs/<task_type>/
```

配置不应存放：

- API Key；
- GitHub Token；
- 私有模型凭证。

敏感信息通过环境变量或 Secret Manager 注入。

------

## 4.4 `eval/data/`

存放评测数据。

每种任务统一使用：

```text
eval/data/<task_type>/
├── seeds/
└── benchmarks/
```

其中：

### `seeds/`

存放尚未正式冻结的候选数据。

Seed 可以包含：

- 候选 PR；
- 候选 GT；
- Review 评论；
- 后续修复线索；
- 待确认 Severity；
- 待确认 Category；
- 待确认 accepted locations；
- formalization checklist。

Seed 不能直接用于正式评分。

### `benchmarks/`

存放经过完整校验和版本冻结的数据。

正式 Benchmark 必须具备：

- 精确的 `base_sha`；
- 精确的 `head_sha`；
- 正式 GT；
- 正式 Severity；
- 正式 Category；
- 正式 `merge_blocking`；
- 正式 accepted locations；
- 文件 SHA-256；
- Benchmark 版本；
- Rubric 版本；
- Judge 版本。

------

## 4.5 `eval/runtime/`

存放评测运行时产生的数据。

包括：

- Git 仓库缓存；
- 临时只读 workspace；
- Agent 原始输出；
- RunTrace；
- Adjudication Table；
- Metrics；
- Markdown/JSON 报告。

这些内容具有以下特征：

- 数量可能很大；
- 可以重新生成；
- 不属于源码；
- 不应默认提交到 Git。

因此整个目录应加入 `.gitignore`，仅保留必要的 `.gitkeep`。

------

# 6. Benchmark 数据分层

## 6.1 正式评分数据

正式评分数据包括：

```text
manifest.yaml
items/*.yaml
```

Benchmark Loader 只需要读取这两类文件。

------

## 6.2 私有构建证据

以下内容放在 `private/`：

- GitHub Review comments；
- Review verdict；
- Review 后修复 commit；
- Review 后讨论；
- GT Jury 原始输出；
- Clean Certification Jury 输出；
- 去重过程；
- 候选 GT；
- 数据来源和生成过程。

这些内容用于：

- 审计 GT；
- 重新运行 Jury；
- 调查争议；
- 升级 Benchmark；
- 解释某条 GT 的来源。

Agent 不得访问 `private/`。

------

## 6.3 Agent 可见数据

Agent 不应直接读取 Benchmark Item 文件。

Runner 从 Item 中提取并构造 Agent Input，仅向 Agent 提供：

- repository；
- PR number；
- title；
- body；
- base branch；
- base SHA；
- head SHA；
- commits；
- changed files；
- PR diff；
- linked issue；
  -固定 SHA 的只读仓库。

以下字段不能传给 Agent：

- `expected_verdict`；
- `clean_status`；
- `gt_findings`；
- `invalidated`；
- Benchmark split；
- Judge 结果；
- Review comments；
- Review 后修复信息。

------

# 7. `manifest.yaml` 正式结构

## 7.1 作用

`manifest.yaml` 是一个 Benchmark 版本的入口文件，负责：

1. 标识 Benchmark 版本；
2. 标识使用的 Rubric 和 Judge 版本；
3. 列出所有 Item；
4. 校验每个 Item 的文件完整性；
5. 保证同一个 Benchmark 版本对应固定的数据内容。

------

## 7.2 正式示例

```yaml
schema_version: pr-review-manifest-v0.1

benchmark_version: pr-review-pilot-v0.1.0-dev
rubric_version: pr-review-rubric-v0.1
judge_version: pr-review-jury-v0.1

created_at: "2026-07-16T00:00:00+00:00"

entries:
  - benchmark_id: pr-review-vllm-omni-3094
    item: items/pr-review-vllm-omni-3094.yaml
    sha256: "a12f03f34cb5c43cfa64a0a229a51accc56ee42a2215ed3e07a0aa823510f731"

  - benchmark_id: pr-review-vllm-omni-2360
    item: items/pr-review-vllm-omni-2360.yaml
    sha256: "79f4368913daf452c253532932c1379b487f3708e3c052d542c7ef73fd373145"

  - benchmark_id: pr-review-vllm-omni-clean-001
    item: items/pr-review-vllm-omni-clean-001.yaml
    sha256: "d8919de83a264d2658db650552ec5783c88d08582f34d8030e5acee3fec37308"
```

------

## 7.3 字段定义

### `schema_version`

```yaml
schema_version: pr-review-manifest-v0.1
```

表示 Manifest 文件结构版本。

修改以下内容时必须升级：

- 字段结构；
- 必填字段；
- 数据类型；
  -校验规则。

它不同于 Benchmark 数据版本。

------

### `benchmark_version`

```yaml
benchmark_version: pr-review-pilot-v0.1.0-dev
```

推荐格式：

```text
<task>-<dataset-name>-v<major>.<minor>.<patch>-<split>
```

例如：

```text
pr-review-pilot-v0.1.0-dev
pr-review-pilot-v0.1.0-test
pr-review-benchmark-v0.2.0-dev
```

------

### `rubric_version`

```yaml
rubric_version: pr-review-rubric-v0.1
```

标识：

- Severity 规则；
- Category 规则；
- Finding Match 规则；
- Validity 规则；
- Verdict 规则；
- 指标规则。

Rubric 语义变化后，不能继续使用相同版本。

------

### `judge_version`

```yaml
judge_version: pr-review-jury-v0.1
```

标识：

- Judge 配置；
- 模型组成；
- Prompt；
- 位置交换规则；
- 第一轮门槛；
- 第二轮门槛；
- 重试策略。

------

### `created_at`

必须使用带时区的 ISO 8601 时间：

```yaml
created_at: "2026-07-16T00:00:00+00:00"
```

------

### `entries`

每个 Entry 对应一个 Benchmark Item：

```yaml
entries:
  - benchmark_id: pr-review-vllm-omni-3094
    item: items/pr-review-vllm-omni-3094.yaml
    sha256: "..."
```

字段含义：

| 字段           | 含义                                 |
| -------------- | ------------------------------------ |
| `benchmark_id` | Item 的全局稳定 ID                   |
| `item`         | 相对于 Manifest 所在目录的 Item 路径 |
| `sha256`       | Item 文件内容的 SHA-256              |

------

## 7.4 Manifest 校验规则

必须满足：

1. `entries` 中的 `benchmark_id` 不得重复；
2. `item` 必须是相对路径；
3. `item` 不得包含 `..`；
4. `item` 必须位于当前 Benchmark 目录内；
5. Item 文件必须存在；
6. Item 内部的 `benchmark_id` 必须与 Entry 一致；
7. Item 文件 SHA-256 必须与 Manifest 一致；
8. 所有 Item 必须通过 Item Schema 校验；
9. 同一个冻结版本中不得静默修改 Item；
10. 修改任何 Item 后必须重新计算 SHA-256，并升级相应 Benchmark 版本。

------

# 8. 正式 PR Review Item 结构

## 8.1 完整示例

```yaml
schema_version: pr-review-item-v0.1

benchmark_id: pr-review-vllm-omni-3094

repository: vllm-project/vllm-omni
pr_number: 3094

base_branch: main
base_sha: "3f5e1a92807c63f968d7436eb43da8b857923ac1"
head_sha: "b9ad822ca71b3d8ed8945f8fea5cebafc7c72a24"

title: "Fix image generation response body"
body: |
  This PR fixes the image generation API response body.

commits:
  - sha: "b9ad822ca71b3d8ed8945f8fea5cebafc7c72a24"
    message: "Fix image generation response body"

changed_files:
  - vllm_omni/entrypoints/openai/api_server.py
  - vllm_omni/entrypoints/openai/protocol/images.py

linked_issue:
  number: 3012
  title: "Image generation response is incompatible with OpenAI API"
  body: |
    The response should use PNG by default.

expected_verdict: REQUEST_CHANGES
clean_status: buggy
split: dev

gt_findings:
  - id: GT-001

    summary: "Omitted output_format changes the documented default encoding"

    description: |
      When output_format is omitted, the implementation selects JPEG.
      The public API contract declares PNG as the default, so callers that
      omit the field receive a different encoding from the documented result.

    severity: Major
    category: compatibility_api
    merge_blocking: true
    location_required: true

    accepted_locations:
      - file: vllm_omni/entrypoints/openai/api_server.py
        start_line: 1545
        end_line: 1555
        symbol: generate_images

    evidence:
      - file: vllm_omni/entrypoints/openai/api_server.py
        start_line: 1538
        end_line: 1562
        reason: |
          The format-selection path does not apply the documented PNG
          fallback when the request omits output_format.

  - id: GT-002

    summary: "The response model rejects size=None"

    description: |
      When the request omits size, the response builder explicitly passes
      null into a field declared as a string. Under Pydantic v2 this causes
      response-model validation to fail instead of returning a valid response.

    severity: Blocker
    category: correctness
    merge_blocking: true
    location_required: true

    accepted_locations:
      - file: vllm_omni/entrypoints/openai/api_server.py
        start_line: 1570
        end_line: 1585
        symbol: generate_images

    evidence:
      - file: vllm_omni/entrypoints/openai/api_server.py
        start_line: 1565
        end_line: 1590
        reason: |
          The response construction includes size even when the request value
          is null, while the response schema requires a string.

invalidated: false
```

# 9. Item 顶层字段定义

## 9.1 `schema_version`

```yaml
schema_version: pr-review-item-v0.1
```

表示 PR Review Item Schema 版本。

------

## 9.2 `benchmark_id`

```yaml
benchmark_id: pr-review-vllm-omni-3094
```

要求：

- 在当前 Benchmark 中唯一；
- 在后续版本中保持稳定；
- 不使用会变化的标题；
- 推荐包含任务、仓库简称和 PR 编号。

推荐格式：

```text
pr-review-<repository-short-name>-<pr-number>
```

Clean PR 也使用真实 PR 编号，不建议使用完全虚构的序号。

例如：

```text
pr-review-vllm-omni-2842
```

其 Clean 状态由 `clean_status` 表示，而不是通过 ID 表示。

------

## 9.3 Repository 和 PR 信息

```yaml
repository: vllm-project/vllm-omni
pr_number: 3094
base_branch: main
base_sha: "..."
head_sha: "..."
```

约束：

- `base_sha` 和 `head_sha` 必须是完整 Git SHA；
- `head_sha` 必须对应 Review 前问题仍然存在的快照；
- 不得使用 Review 后修复 commit；
- 两个 SHA 都必须能够从冻结的仓库快照恢复。

------

## 9.4 PR 文本信息

```yaml
title: "..."
body: "..."
```

应保存固定快照时可见的 PR 标题和正文。

不能使用 Review 后才修改出的说明作为 Agent 输入。

------

## 9.5 `commits`

```yaml
commits:
  - sha: "..."
    message: "..."
```

只包含截至 `head_sha` 可见的 commit。

不得包含：

- Review 后修复 commit；
- 最终合并 commit；
- 后续 rebase commit；
- Benchmark 构建期间新增的内部 commit。

------

## 9.6 `changed_files`

```yaml
changed_files:
  - vllm_omni/entrypoints/openai/api_server.py
```

必须与冻结的 `base_sha..head_sha` diff 一致。

建议只保存路径，具体 diff 在运行时通过 Git 快照生成，避免 YAML 中保存重复的大体积 diff。

------

## 9.7 `linked_issue`

可选字段：

```yaml
linked_issue:
  number: 3012
  title: "..."
  body: "..."
```

只有 PR 明确引用的 Issue 才能写入。

不得通过 Benchmark 构建者主观关联一个 Agent 原本不可见的 Issue。

------

## 9.8 `expected_verdict`

只能是：

```text
APPROVE
REQUEST_CHANGES
```

计算规则：

```text
存在至少一个 merge_blocking=true 的 GT Finding
    → REQUEST_CHANGES

否则
    → APPROVE
```

`expected_verdict` 必须由 GT 推导，并在 Schema 校验时验证一致性。

------

## 9.9 `clean_status`

允许值：

```text
buggy
auto_certified_clean
```

### Buggy PR

```yaml
clean_status: buggy
```

通常至少包含一个有效 GT Finding。

### Auto-certified Clean PR

```yaml
clean_status: auto_certified_clean
expected_verdict: APPROVE
gt_findings: []
```

只有完成三个独立 Reviewer Agent 和 Clean Certification Jury 后，才能标记为 `auto_certified_clean`。

------

## 9.10 `split`

允许：

```text
dev
test
```

含义：

- `dev`：允许开发人员查看 GT；
- `test`：GT 对 Agent 开发流程隐藏，只允许正式评测系统访问。

虽然 Item 中包含 `split`，Dev 和 Test 仍应使用物理上不同的 Benchmark 目录。

------

## 9.11 `invalidated`

```yaml
invalidated: false
```

当 Auto-certified Clean PR 上出现经增强 Jury 确认的 `VALID_NEW` 时：

```yaml
invalidated: true
```

该 Item 应在新 Benchmark 版本中统一排除或重新标注。

冻结版本中的文件不应原地修改，应通过新版本表达 invalidation。

------

# 10. GT Finding 结构

## 10.1 基本字段

```yaml
id: GT-001
summary: "..."
description: "..."
severity: Major
category: correctness
merge_blocking: true
location_required: true
```

------

## 10.2 `id`

一个 Item 内唯一：

```text
GT-001
GT-002
GT-003
```

同一个 GT 在 Benchmark Patch 版本中不应随意更换 ID。

------

## 10.3 `summary`

一句话概括问题根因或核心错误。

应描述：

- 真正的问题；
- 而不是 Reviewer 的措辞；
- 也不是后续修复方式。

正确示例：

```text
The cache key omits request-specific state
```

不推荐：

```text
Please update this line
```

------

## 10.4 `description`

必须说明：

1. 问题是什么；
2. 什么条件下触发；
3. 会造成什么影响；
4. 为什么当前代码存在该问题。

不能直接复制完整历史 Review comment 作为 GT。

------

## 10.5 `severity`

允许：

```text
Critical
Blocker
Major
Minor
Nit
```

约束：

| Severity | `merge_blocking`         |
| -------- | ------------------------ |
| Critical | 必须为 `true`            |
| Blocker  | 必须为 `true`            |
| Major    | 可以为 `true` 或 `false` |
| Minor    | 必须为 `false`           |
| Nit      | 必须为 `false`           |

------

## 10.6 `category`

只能选择一个 Primary Category：

```text
correctness
compatibility_api
concurrency
performance_resource
security_safety
test
documentation
maintainability
```

不得为一个 Finding 设置多个 Primary Category。

------

## 10.7 `merge_blocking`

```yaml
merge_blocking: true
```

表示问题是否必须在当前 PR 合并前修复。

它不完全由 Severity 推导：

- Critical、Blocker 必须 blocking；
- Minor、Nit 必须 non-blocking；
- Major 由 Jury 独立判断。

------

## 10.8 `location_required`

```yaml
location_required: true
```

表示该问题是否要求 Agent 给出代码定位。

通常代码缺陷为 `true`。

以下问题可能是 `false`：

- PR 整体缺少必要设计说明；
- 跨大量文件且不存在单一根因位置；
- 纯 PR 级策略问题。

------

# 11. Accepted Location 结构

示例：

```yaml
accepted_locations:
  - file: vllm_omni/model.py
    start_line: 120
    end_line: 138
    symbol: build_model
```

字段：

| 字段         | 必填 | 含义                        |
| ------------ | ---- | --------------------------- |
| `file`       | 是   | `head_sha` 中的仓库相对路径 |
| `start_line` | 否   | 起始行                      |
| `end_line`   | 否   | 结束行                      |
| `symbol`     | 否   | 函数、类或其他代码符号      |

规则：

1. `file` 必须在 `head_sha` 中存在；
2. 行号必须处于文件范围内；
3. `start_line <= end_line`；
4. 至少提供行区间或 symbol；
5. 可以设置多个 accepted locations；
6. accepted location 必须指向根因或直接导致问题的代码；
7. 不应只指向后续修复代码。

------

# 12. Evidence 结构

示例：

```yaml
evidence:
  - file: vllm_omni/model.py
    start_line: 110
    end_line: 150
    reason: |
      The new branch returns before the required state is initialized.
```

Evidence 用于 Judge 和 Benchmark 审计。

规则：

1. 必须来自固定的 `base_sha/head_sha`；
2. 不得引用 Review 后代码；
3. 不得直接包含历史 Reviewer 的最终判断；
4. `reason` 应解释该代码如何支持 GT；
5. Evidence 可以包含定义、调用方和相关测试源码；
6. 更完整的大体积证据可以存入 `private/evidence/`，Item 中只保留关键索引。

------

# 13. Seed 与正式 Benchmark 的转换

Seed 目录：

```text
eval/data/pr_review/seeds/pr-review-pilot-seed-v0.1/
```

正式 Benchmark：

```text
eval/data/pr_review/benchmarks/pr-review-pilot-v0.1.0-dev/
```

转换过程：

```text
Seed 候选 PR
    ↓
补充 base_sha/head_sha
    ↓
恢复完整 Git 快照
    ↓
构建证据包
    ↓
执行 GT Jury
    ↓
执行去重
    ↓
冻结 Severity、Category、merge_blocking
    ↓
冻结 accepted locations
    ↓
生成正式 Item
    ↓
计算 Item SHA-256
    ↓
生成 manifest.yaml
    ↓
执行完整 Schema 校验
    ↓
冻结 Benchmark 版本
```

Seed 中以下字段不能直接进入正式 Item：

```text
status
pr_url
snapshot_rule
post_review_fix_sha
expected_verdict_candidate
candidate_gt_findings
severity_candidate
category_candidate
merge_blocking_candidate
accepted_locations_candidate
remaining_work
```

这些信息应移动到：

```text
private/github/
private/jury/
private/provenance/
```

# 16. CLI 路径约定

  --baseline-run <baseline-run-id> \
  --candidate-run <candidate-run-id> \

# 19. 最终目录结论

正式采用：

```text
eval/
├── common/       # 跨任务共享代码
├── tasks/        # 各种评测任务代码
├── configs/      # Agent、Judge 和工具配置
├── data/         # Seed 和冻结 Benchmark
└── runtime/      # 缓存、workspace、运行结果和报告
```

PR Review 对应：

```text
评测代码：
eval/tasks/pr_review/

Seed：
eval/data/pr_review/seeds/pr-review-pilot-seed-v0.1/

正式 Dev Benchmark：
eval/data/pr_review/benchmarks/pr-review-pilot-v0.1.0-dev/

正式 Test Benchmark：
eval/data/pr_review/benchmarks/pr-review-pilot-v0.1.0-test/
```

其中正式 Test Benchmark 只存在于私有评测环境，不进入公开仓库。

每个正式 Benchmark 版本至少包含：

```text
manifest.yaml
items/*.yaml
```

推荐同时保存：

```text
private/github/
private/evidence/
private/jury/
private/provenance/
```

Manifest 负责版本、Item 索引和文件完整性校验；Item 负责冻结 PR 快照、GT、Severity、Category、merge-blocking、accepted locations 和 expected verdict。

该结构可以在不增加项目根目录复杂度的前提下，稳定扩展到后续所有 Copilot 评测任务。
